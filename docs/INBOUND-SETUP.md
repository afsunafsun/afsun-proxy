# Настройка входящих подключений и клиентов (3X-UI)

Справочник по полям панели на русском. Для продажи: **один инбаунд** → много **клиентов**.

---

## Схема

```
Входящее подключение (inbound)     Клиент (client)
├── протокол: VLESS / SOCKS        ├── email (логин в панели)
├── порт: 443 / 10808              ├── UUID / пароль
├── Reality / TLS                  ├── лимит трафика, срок
└── настройки транспорта           └── подписка / QR
```

**Порты в UFW:** панель **не открывает** файрвол. После создания inbound:

```bash
sudo bash scripts/sync-ufw-inbounds.sh
```

Базовые порты — `scripts/setup-ufw.sh`. Подробно: [SECURITY.md](SECURITY.md).

## VLESS + Reality (VPN для телефона) — рекомендуется

### Шаг 1: Входящее подключение

**Входящие подключения** → **Создать входящее подключение**

| Поле в панели | Что выбрать | Зачем |
|---------------|-------------|--------|
| **Примечание** | `VLESS Reality 443` | Имя для себя |
| **Протокол** | `vless` | VPN-протокол |
| **Порт** | `443` | Уже открыт в UFW; стандартный HTTPS-порт |
| **Включить** | да | |

#### Транспорт

| Поле | Значение | Объяснение |
|------|----------|------------|
| **Протокол** (transport) | `TCP (RAW)` | Обычный TCP, стандарт для Reality |
| **Proxy Protocol** | **выкл** | Вкл. только если перед Xray стоит Nginx/HAProxy с `proxy_protocol`. У нас — нет |
| **HTTP Маскировка** | **выкл** | Для WebSocket-маскировки; с Reality не нужна |
| **Sockopt** | **выкл** | Продвинутые сокеты; по умолчанию не трогать |
| **External Proxy** | **выкл** | Цепочка через другой прокси; не нужно |
| **Безопасность** | **`Reality`** | **Не «Пусто»!** Иначе трафика не будет |

Подробные поля **Reality** и **TLS** — в разделах ниже.

#### Sniffing (на инбаунде)

| Значение | Когда |
|----------|--------|
| **вкл** | VPN: Xray видит домены, лучше маршрутизация |
| **выкл** | Можно для SOCKS |

Обычно для VLESS: **вкл**, destOverride: `http`, `tls`.

### Шаг 2: Клиент (на каждого покупателя)

В строке инбаунда → **Создать клиента**

| Поле | Значение | Объяснение |
|------|----------|------------|
| **Включить** | да | Иначе не подключится |
| **Email** | `user1@shop` | Уникальный ID в панели (не обязательно настоящая почта) |
| **ID** | авто UUID | Уникальный ключ VLESS. Кнопка 🔄 — новый UUID (старая подписка перестанет работать) |
| **Flow** | см. раздел **«Flow»** ниже | Критично для VLESS+Reality |
| **Комментарий** | опционально | Заметка (имя клиента) |
| **Общий расход** | ГБ или `0` | Лимит трафика. `0` = без лимита. Единица — ГБ |
| **Использование** | только чтение | Сколько уже израсходовано; кнопка сброса — обнулить счётчик |
| **Начало использования** | обычно **выкл** | Отложенный старт: трафик/срок начинают идти с первого подключения, а не с момента создания |
| **Дата окончания** | дата или пусто | После даты клиент **отключается** автоматически |
| **Лимит по IP** | `0` | Макс. одновременных IP с одного клиента; `0` = без лимита |

После создания: **Подробнее** → ссылка подписки / QR.

### Flow — три варианта в выпадающем списке

Поле **Flow** видно только у **VLESS** с **TLS** или **Reality**. Для SOCKS не используется.

| Flow в панели | Когда выбирать | Поведение |
|---------------|----------------|-----------|
| **Пусто / none** | Не использовать с Reality | Обычный TLS-прокси **без XTLS**. С Reality на 443 — **не подходит**: нет трафика или обрывы |
| **`xtls-rprx-vision`** | **По умолчанию. Рекомендуем** | XTLS Vision: быстрый туннель, маскировка. **Перехватывает UDP на порт 443** (QUIC) → браузер переключается на TCP HTTPS. Рекомендуется для стабильности |
| **`xtls-rprx-vision-udp443`** | Редко, по необходимости | То же, что Vision, но **не трогает UDP 443** — QUIC (HTTP/3) работает. Нужно, если без QUIC ломаются YouTube, Discord, часть сайтов |

#### Что выбрать на практике (afsun-proxy)

```
VLESS + Reality + TCP  →  flow = xtls-rprx-vision
```

**Не оставлять Flow пустым** — типичная ошибка «подключился, интернета нет».

#### Когда переключить на `xtls-rprx-vision-udp443`

- С `xtls-rprx-vision` сайты на **HTTP/3 / QUIC** не открываются, а на TCP — ок.
- Нужны сервисы, которые **настаивают на UDP 443**.

Минусы udp443: QUIC может хуже проходить через некоторые сети. У нас **3.4.0**: если после смены на udp443 клиент не коннектится — вернуть **`xtls-rprx-vision`** и обновить подписку.

#### Flow на сервере и в приложении

- **Сервер (панель):** flow задаётся **у каждого клиента** в инбаунде.
- **Клиент (v2rayNG / Hiddify):** flow подтягивается из **подписки** (`flow=xtls-rprx-vision` в ссылке). Вручную менять не нужно, если подписка свежая.

#### Симптомы неправильного Flow

| Симптом | Решение |
|---------|---------|
| Подключение есть, трафик 0 | Поставить `xtls-rprx-vision`, обновить подписку |
| Часть сайтов не грузится | Попробовать `xtls-rprx-vision-udp443` |
| После смены Flow не работает | Удалить профиль в приложении → импорт подписки заново |

## SOCKS5 (Telegram, браузер)

**Входящие подключения** → протокол **socks** или **mixed**

| Поле | Значение |
|------|----------|
| **Порт** | `10808` | После создания: `sync-ufw-inbounds.sh` |
| **auth** | `password` для продажи (логин/пароль на клиента) |
| **auth** | `noauth` только для теста |
| **Sniffing** | обычно **выкл** |

Клиент: в панели задаётся **логин/пароль** (если auth включён).

В приложении вручную:

```
Хост: YOUR_DOMAIN
Порт: 10808
```

**Не путать с VPN:** v2rayNG в режиме VPN ждёт VLESS, не SOCKS.

## Безопасность: Reality (рекомендуется для VPN)

В блоке **Безопасность** выбрать **`Reality`** (не «Пусто», не TLS).

Reality маскирует трафик под обычный HTTPS к чужому сайту (yahoo.com, microsoft.com). **Свой SSL-сертификат на 443 не нужен** — порт 443 занимает Xray, Caddy на 8443.

### Публичный и закрытый ключ + «Get New Cert»

Кнопка **Get New Cert** в панели генерирует **пару ключей Reality** (алгоритм **X25519**). Название «Cert» историческое — это **не** SSL-сертификат Let's Encrypt, как у TLS.

#### Зачем они нужны

Reality должен убедиться, что к серверу подключается **ваш клиент**, а не случайный сканер или провайдер. Для этого используется криптографическая пара:

| Ключ | Где живёт | Кто видит |
|------|-----------|-----------|
| **Закрытый ключ (Private)** | **только на сервере** (Xray) | Никому не отдавать, не коммитить, не слать клиентам |
| **Публичный ключ (Public)** | в подписке / ссылке клиента | Можно отдавать каждому покупателю |

**Аналогия:** закрытый ключ — как ключ от замка в двери сервера; публичный — как «метка», по которой клиентское приложение понимает, что это именно ваш сервер Reality, а не поддельная заглушка.

#### Что делать при создании инбаунда

1. Выбрать **Reality**, заполнить **Цель** (`домен:443`), SNI, uTLS.
2. Нажать **Get New Cert** один раз — поля **Публичный ключ** и **Закрытый ключ** заполнятся автоматически.
3. Нажать **Создать** / **Обновить** инбаунд.
4. **Не нажимать Get New Cert повторно** без необходимости — см. ниже.

Ручной ввод ключей не нужен, если панель генерирует их сама.

#### Что попадает к клиенту

В подписке / `vless://` ссылке публичный ключ передаётся как параметр **`pbk=...`** (public key). Пример фрагмента:

```
vless://UUID@YOUR_DOMAIN:443?...&pbk=ТУТ_ПУБЛИЧНЫЙ_КЛЮЧ&sid=SHORT_ID&...
```

Клиенту также нужны: **UUID**, **SNI**, **flow**, **shortId** (один из Short IDs) — всё это панель собирает в ссылку сама.

**Закрытый ключ в ссылку никогда не попадает.** Если утёк — сгенерируй новую пару (Get New Cert) и перевыдай подписки.

#### Если снова нажать Get New Cert

- Старая пара ключей **перестаёт работать**.
- Все клиенты с **старыми подписками** отвалятся.
- Нужно: новая подписка каждому + обновление в v2rayNG/Hiddify.

**Когда менять ключи:** компрометация сервера, утечка private key, плановая ротация раз в N месяцев. Не «на всякий случай» каждый день.

#### Связь с Short IDs

| Параметр | Роль |
|----------|------|
| **Публичный/закрытый ключ** | Криптография Reality — «свой ли это сервер» |
| **Short IDs** | Короткий идентификатор внутри Reality; у одного инбаунда может быть несколько (разные клиенты могут получать разные `sid`) |

Оба обязательны для рабочего подключения. Ключи — **одна пара на инбаунд**; Short IDs — **список**, панель назначает клиенту один из них.

#### Отличие от TLS (чтобы не путать)

| | **Reality: Public / Private key** | **TLS: сертификат .pem** |
|---|-----------------------------------|---------------------------|
| Тип | X25519 keypair | X.509 сертификат + приватный ключ CA |
| Кнопка в панели | **Get New Cert** | загрузка / путь к файлам |
| Нужен свой домен на 443 | нет | да |
| В ссылке клиента | `pbk` | CA / allowInsecure / SNI |

### Поля Reality — что писать

| Поле | Что указать | Зачем |
|------|-------------|--------|
| **Show** | **выкл** | Отладочный режим; в проде не нужен |
| **Xver** | `0` | Версия PROXY protocol внутри Reality; оставить `0` |
| **uTLS** | `firefox` или `chrome` | Отпечаток TLS. Варианты: `chrome`, `firefox`, `safari`, `ios`, `android` |
| **Dest (Target / Цель)** | `www.yahoo.com:443` | **Обязательно** `домен:443`. В 3X-UI 3.4.0 поле называется **Цель** (`target`). Пустое = сломается при сохранении в панели. Скрипты проставляют автоматически |
| **SNI** | `www.yahoo.com`, `yahoo.com` | Имена в TLS ClientHello. Обычно совпадают с доменом из Цели |
| **Max Time Diff (ms)** | `0` | Допуск рассинхрона часов клиент/сервер. `0` = по умолчанию |
| **Short IDs** | оставить сгенерированные | Короткие hex-ID для клиентов, через запятую. Кнопка 🔄 — перегенерировать. **Не менять** после выдачи подписок без обновления у клиентов |
| **SpiderX** | `/` | Путь для «паутины» Reality при проверке dest. Почти всегда `/` |
| **Публичный ключ** | **Get New Cert** | Уходит клиентам в подписке (`pbk`). См. раздел выше |
| **Закрытый ключ** | **Get New Cert** | Только сервер. Не светить, не копировать в чат клиентам |

### Пример готовой конфигурации Reality

```
uTLS:      firefox
Цель:      www.yahoo.com:443
SNI:       www.yahoo.com, yahoo.com
Short IDs: (сгенерированные панелью, не трогать после выдачи подписок)
SpiderX:   /
Ключи:     Get New Cert → оба поля заполнены → Создать/Обновить
Клиент:    flow = xtls-rprx-vision, подписка из «Подробнее»
```

### У клиента (обязательно)

| Поле | Значение |
|------|----------|
| **Flow** | **`xtls-rprx-vision`** (см. раздел **Flow** в «Шаг 2: Клиент») |

Без flow — подключение есть, **трафика нет**.

### Когда менять Цель / SNI

Если timeout при рабочем сервере — другой target: `www.cloudflare.com:443` или `sudo bash scripts/tune-reality-settings.sh`.  
Если в панели **Цель пустая**, а VPN работал — срочно: `sudo bash scripts/fix-reality-target.sh` (иначе re-save в панели сломает Reality).

## Безопасность: TLS (альтернатива Reality)

Выбирать **`TLS`**, если нужен **свой домен + сертификат** на inbound (например VLESS на поддомене с Let's Encrypt).

**У нас по умолчанию:** Caddy уже держит TLS на **8443** для панели; для VPN на **443** проще **Reality**. TLS на 443 имеет смысл, если отдельно настроите сертификат для `YOUR_DOMAIN` на этом порту (сложнее, конфликт с Reality).

### Поля TLS — что писать

| Поле | Что указать | Зачем |
|------|-------------|--------|
| **SNI** | `YOUR_DOMAIN` | Домен в сертификате. Должен совпадать с CN/SAN сертификата |
| **Cipher Suites** | `Auto` | Набор шифров; оставить Auto |
| **Min Version** | `1.2` | Минимум TLS 1.2 |
| **Max Version** | `1.3` | Максимум TLS 1.3 |
| **uTLS** | `chrome` | Fingerprint для клиента (если используется) |
| **ALPN** | `h2`, `http/1.1` | Протоколы поверх TLS. Обычно `h2` + `http/1.1`; `h3` — если настроен HTTP/3 |
| **Allow Insecure** | **выкл** | Не проверять сертификат — только для тестов |
| **Reject Unknown SNI** | **выкл** | Отклонять чужой SNI; для одного домена можно **вкл** |
| **Disable System Root** | **выкл** | Не использовать системные CA |
| **Session Resumption** | по желанию | Ускорение повторных подключений TLS |
| **VerifyPeerCertInNames** | пусто или DNS | Для исходящих проверок; для inbound обычно не трогать |
| **Цифровой сертификат** | **Путь к сертификату** | Вкладка «Путь» — если сертификаты на диске |

### Сертификат (путь или содержимое)

Здесь **Публичный ключ** и **Закрытый ключ** — это уже **файлы SSL/TLS** (PEM), не Reality X25519.

| Поле | Что указать |
|------|-------------|
| **Публичный ключ** | `/path/to/fullchain.pem` — цепочка сертификата (сервер + промежуточные CA) |
| **Закрытый ключ** | `/path/to/privkey.pem` — приватный ключ домена |

Вкладка **«Содержимое сертификата»** — вставить текст PEM вручную, если нет файлов на диске.

**Порядок действий для TLS:**

1. Получить сертификат на домен (Let's Encrypt, Caddy, certbot).
2. Указать пути или вставить PEM в оба поля.
3. **SNI** = тот же домен, что в сертификате.
4. Сохранить инбаунд, проверить с клиента.

Закрытый ключ TLS — тоже **только на сервере**, как и у Reality.

Пример путей (если сертификат Let's Encrypt на хосте):

```
Публичный:  /var/lib/afsun-proxy/caddy/data/.../fullchain.pem
Закрытый:     /var/lib/afsun-proxy/caddy/data/.../privkey.pem
```

На практике проще не дублировать: **VPN = Reality на 443**, **панель = Caddy на 8443**.

### Reality vs TLS — что выбрать

| | **Reality** | **TLS** |
|---|-------------|---------|
| Сертификат на 443 | не нужен | нужен свой |
| Маскировка | под чужой HTTPS-сайт (yahoo.com и т.д.) | под ваш домен |
| Для VPN на 443 | **рекомендуем** | если нужен свой сертификат |
| Настройка | Get New Cert + dest | пути к .pem |

## Что означают опции (краткий словарь)

| Опция | За что отвечает |
|-------|-----------------|
| **TCP (RAW)** | Трафик как есть, без WebSocket/gRPC |
| **WebSocket / gRPC / XHTTP** | Другие транспорты; для Reality обычно не нужны |
| **Пусто** (безопасность) | Без шифрования TLS/Reality — для VLESS на 443 **не использовать** |
| **TLS** | Классический HTTPS с сертификатом на вашем домене |
| **Reality** | Маскировка под чужой сайт (microsoft.com и т.д.), без своего сертификата на 443 |
| **Proxy Protocol** | Передача реального IP клиента от балансировщика; для прямого доступа **выкл** |
| **HTTP Маскировка** | Путь/host как у обычного HTTP-сайта |
| **Sockopt** | TCP keepalive, MPTCP и пр. — для тонкой настройки |
| **External Proxy** | Выход в интернет через другой прокси (цепочка) |
| **Fallbacks** | Запасные маршруты при ошибке — для Reality редко нужны |
| **Flow `xtls-rprx-vision`** | XTLS Vision; **по умолчанию** для Reality |
| **Flow `xtls-rprx-vision-udp443`** | Vision + разрешён QUIC (UDP 443) |
| **Flow пустой** | Без XTLS; с Reality **не использовать** |
| **Sniffing** | Определение домена внутри соединения для правил маршрутизации |
| **Fragment / Mux** | В настройках клиента — фрагментация TLS при проблемах связности |

## Чеклист перед выдачей клиенту

- [ ] Порт инбаунда открыт в UFW: `sudo bash scripts/sync-ufw-inbounds.sh`
- [ ] Xray **Запущен** (дашборд)
- [ ] VLESS: **Reality** + **flow** на клиенте
- [ ] Клиент **включён**, срок/лимит заданы
- [ ] Подписка обновлена в приложении (старый профиль удалён)
- [ ] Не работает → [TROUBLESHOOTING.md](TROUBLESHOOTING.md)

## Типичные ошибки

| Симптом | Причина |
|---------|---------|
| Timeout | Порт не в UFW / нет flow / см. TROUBLESHOOTING |
| Подключился, трафика 0 | Нет `xtls-rprx-vision` или не Reality |
| SOCKS не работает | Порт 24413 вместо 10808 / VPN-режим вместо proxy |
| Панель есть, VPN нет | Создан только SOCKS, нет инбаунда VLESS |
